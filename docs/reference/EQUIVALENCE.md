# PLY/SPZ equivalence contract

Both bundles decode into the same format-independent
[`GaussianCloudData`](GAUSSIAN_MODEL_CONTRACT.md). This document defines what
"equivalent" means when one source model is encoded into both formats, and
derives the tolerances the cross-format test enforces.

The test is `tests/equivalence/test_equivalence.cpp`; the paired fixtures are
generated by `tools/generate_equivalence_fixtures.py`.

## 1. Why this test exists

The PLY and SPZ decoders are independent implementations that must land on the
same model. Several claims are asserted nowhere else:

- the RUB→RDF reference-frame conversion ([SPZ_MAPPING.md §5](SPZ_MAPPING.md)) —
  PLY applies no conversion, so PLY *is* the reference frame;
- the 15 per-coefficient SH band sign flips;
- the two spherical-harmonic memory layouts — PLY stores rest coefficients
  channel-major ([PLY_MAPPING.md §4](PLY_MAPPING.md)), SPZ stores them
  Gaussian-major ([SPZ_MAPPING.md §6](SPZ_MAPPING.md)) — reaching one
  canonical order;
- the quaternion component order and the two SPZ rotation encodings.

The per-bundle decoder suites check each format against re-derived formulas or
an encode→decode inverse. Those catch a wrong formula; they cannot catch a
formula that is wrong *consistently* on both the encode and decode side. The
cross-format comparison can, because the other format is an independent
witness.

## 2. Tolerance derivation

The fixture generator defines Gaussians in shared-model space and encodes each
one twice. The PLY encoding is exact at float32 — positions, quaternions, and
SH coefficients are stored as-is, and only `log`/`exp` and `logit`/`sigmoid`
round-trips introduce error, at relative 1e-7. **The entire disagreement budget
therefore belongs to SPZ quantization**, and each tolerance is half of one SPZ
quantization step:

| Attribute | SPZ storage | Step | Tolerance |
| --- | --- | --- | --- |
| position (v2/v3) | 24-bit fixed point, `fractionalBits = 12` | 2⁻¹² | 2⁻¹³ ≈ 1.22e-4 |
| position (v1) | float16 | value-dependent | not paired; see §5 |
| scale | `uint8`, log-encoded, `exp(b/16 - 10)` | 1/16 in log space | relative `exp(1/32) - 1` ≈ 3.18e-2 |
| opacity | `uint8`, `b/255` | 1/255 | 1/510 ≈ 1.96e-3 |
| DC | `uint8`, `(b/255 - 0.5)/0.15` | 1/38.25 | 1/76.5 ≈ 1.31e-2 |
| SH rest | `uint8`, `(b - 128)/128` | 1/128 | 1/256 ≈ 3.91e-3 |
| rotation | see below | see below | 8e-3 |

Scale is the one *relative* tolerance: SPZ quantizes the logarithm, so the
absolute error grows with the value.

Rotation is allowed more than a half step. Both SPZ encodings omit one
component and reconstruct it as `sqrt(1 - Σ)`, whose error is amplified by
`1/(2w)` as the reconstructed component approaches zero. The fixtures keep the
reconstructed component well away from zero, which bounds the amplification;
8e-3 then covers `first-three` (step 1/127.5) with margin, and `smallest-three`
(step `sqrt(1/2)/511`) comfortably.

## 3. Two tolerance profiles

Holding an on-grid fixture to the envelope above would let a real structural
error hide inside a tolerance three orders of magnitude looser than the actual
disagreement. So the fixtures come in two kinds:

- **On-grid (`kExact`, 1e-5).** Every attribute except rotation sits exactly on
  an SPZ quantization point, so quantization contributes nothing and only
  float32 rounding remains. This is the sharp structural check.
- **Off-grid (`kEnvelope`).** Arbitrary values chosen *between* quantization
  points, asserting the §2 envelope holds for input that was not selected to
  round-trip.

Rotation uses the envelope value in both profiles: `first-three` stores `w`
implicitly, so no quaternion round-trips exactly — not even the identity.

Values sitting exactly on a half-step boundary are avoided deliberately. At the
boundary the encoder's rounding rule decides the result, so the measured error
would land exactly on the tolerance and the assertion would be a coin flip.

## 4. The SH quantization range is asymmetric

SPZ dequantizes rest coefficients as `(b - 128)/128` with `b` in `0..255`, so
the representable range is **`[-1.0, +0.9921875]`** — not symmetric. Byte 0 is
the one grid point whose negation is *not* on the grid.

This matters because the RUB→RDF conversion multiplies coefficients by ±1: a
coefficient of `-1.0` carrying a negative band flip is encoded as `+1.0`, which
clamps to `+0.9921875`. That is a property of the SPZ encoding, not a decoder
disagreement, and it is why the on-grid fixtures draw their SH bytes from
`1..255`. A real asset can still contain such a value; the resulting error is
bounded by one full step (1/128) and is attributable to the encoder that wrote
the file.

## 5. What is deliberately not compared

- **v1 float16 positions.** The v1 position path has a value-dependent step and
  is pinned by the SPZ decoder suite against known values instead. Pairing it
  would mean either a tolerance loose enough to be meaningless or fixtures
  restricted to exactly representable float16 values, which tests less than the
  existing v1 fixture already does.
- **The real-asset corpus.** `yashica-t4` and `leica-sofort` are independent
  captures, not two encodings of one reconstruction; the PLY and SPZ corpora
  must never be diffed against each other. See
  [SPZ_FORMAT.md §8](SPZ_FORMAT.md).
- **The authored USD stage.** Both bundles author through the single
  `libs/gaussian-usd` writer, so hierarchy, schema, metadata policy, stage
  metrics, and default-prim behavior are identical by construction and are
  asserted per-bundle by each smoke test. The only intended difference is the
  `sourceFormat` string.

## 6. Where the test runs

`tests/equivalence/` links both bundles' decoder libraries, which no single
bundle can do. It is configured only by the plain root CMake composition —
`ost plugin build <bundle>` configures one bundle and never reaches it — and
runs as the `gaussian_ply_spz_equivalence` ctest.
