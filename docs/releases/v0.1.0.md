# v0.1.0 — first tagged release

**Tagged:** 2026-07-19 (`c51eb29`) · **Published:** draft, pending human review ·
[Release page](https://github.com/animu-sphere/usd-3dgs-plugins/releases/tag/v0.1.0)

The first tagged release: a read-only Gaussian Splatting PLY importer that
decodes the canonical Graphdeco dialect and authors OpenUSD 26.05's standard
`ParticleField3DGaussianSplat` schema, plus the release machinery around it.

## Objective

Turn the locally green Gaussian PLY vertical slice into an *observed*, portable,
reproducibly packaged release across the declared support matrix — proving the
PLY-to-USD semantic mapping on real hosted runners rather than one developer
machine, before any renderer or second format exists.

## What shipped

- **Importer** (`SdfFileFormat` for `ply`): ASCII and binary little-endian
  decoding through vendored tinyPLY 2.3.4, Gaussian dialect detection, and
  required-property validation.
- **Semantic conversion:** position, `exp(stored)` scale, scalar-first
  normalized quaternion, stable-sigmoid opacity, and spherical-harmonic
  reconstruction with degree inference for channel-major `f_rest_*`.
- **Stage contract:** `/Asset` Xform (`kind=component`, defaultPrim) with
  `/Asset/Splat` as `ParticleField3DGaussianSplat`; `upAxis = Y` and
  `metersPerUnit = 1` as documented importer policy, since PLY standardizes
  neither.
- **Format-independent core:** `gaussianCore` carries the model, math, and
  validation with no USD types, described by an OpenStrata library descriptor.
- **Verification:** deterministic C++, Python, negative, binary, and OST golden
  tests, run on hosted Windows, macOS arm64, and Linux.
- **Release machinery:** a tag-driven lane deriving its matrix from
  `openstrata.ci.yaml`, gating on version agreement and a finalized changelog,
  and proving digest-reproducible packaging before assembling a draft.

Granular detail: [delivery history](../reports/delivery-history.md).

## Compatibility

- **OpenUSD:** cy2026 / OpenUSD 26.05 (the tolerated range is `>=26.05,<27.0`;
  the `ParticleField3DGaussianSplat` schema requires 26.05)
- **Platforms:** Windows x86_64 (msvc143), macOS arm64 (libcxx),
  Linux x86_64 (libstdcxx)
- **Python ABI:** 3.13
- **Built with:** OST 0.18.0

A binary target must match the host OpenUSD build, platform/compiler ABI, and
Python ABI.

## Known limitations at this release

- **Multi-million-Gaussian PLYs fail to load.** The v0.1.0 read path
  serializes the generated stage to USDA text and reparses it; at large
  Gaussian counts (observed 2026-07-19 with the 5.8M-splat Mip-NeRF 360
  `garden` reference: `[GSPLY]`-less error "generated USD for '…' could not
  be parsed") the multi-gigabyte text round-trip fails and the stage does not
  open. Fixed after v0.1.0 by authoring the layer directly and transferring
  its content (commit `0f34f7a`); ships in v0.2.0.
- Read-only, fully materialized PLY import; no writing and no streaming.
- Canonical Graphdeco property names only. `f_rest_*` indices must be
  contiguous, and with `R` rest scalars `R / 3 + 1` must be a square count.
- No bundled Gaussian renderer — this project authors Gaussian USD data. A
  stage opens even when the active Hydra renderer cannot draw splats.
- Windows verified to OST L4 rather than L5 (see above).
- Package-origin L5 **skips rather than executes**: OST 0.18.0 packages the
  roundtrip PLY fixture but not its adjacent `.golden.usda`, and the bundle
  manifest has no golden declaration. Upstream packaging seam, recorded in
  [dogfooding report 01](../reports/ost/01-2026-07-18-v0.18.0-bootstrap.md).
- The extracted package was not exercised from a clean directory outside the
  worktree, and manual OpenUSD activation on Windows was not verified; only the
  `ost plugin run <package>` path was.
- No real-asset corpus. Every fixture is synthetic and small, so decode time,
  peak memory, and output array sizes at realistic Gaussian counts are
  unmeasured.
- SPZ, glTF/GLB Gaussian extensions, and SOG are out of scope for this release.

## Notes on the release itself

The release lane was exercised by two `workflow_dispatch` dry runs before the
tag. The first surfaced a defect in the checksum step: `SHA256SUMS` listed
itself, hashed as the empty string, because the `> SHA256SUMS` redirection
raced the `find` that enumerated the directory. It passed locally, where `find`
won that race, and failed on the runner, where it lost. Left unfixed it would
have made `sha256sum -c` fail for every consumer of this release. The lane now
excludes the name explicitly and verifies the manifest with `sha256sum -c`
before publishing it.
