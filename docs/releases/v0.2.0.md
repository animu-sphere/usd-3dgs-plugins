# v0.2.0 — production-ready Graphdeco PLY import

**Tagged:** 2026-07-19 (`7323f46`) · **Published:** draft, pending human review ·
[Release page](https://github.com/animu-sphere/usd-3dgs-plugins/releases/tag/v0.2.0)

The release-plan Phase 1 stabilization theme: complete and stabilize Graphdeco
PLY support before adding another format. Failure modes are now diagnosable by
stable identifier, import behavior is configurable per asset, performance is
measured rather than assumed, and multi-million-Gaussian captures load.

## Objective

Turn the v0.1.0 vertical slice into an importer that holds up against real
exporter output at real capture sizes: fix the large-asset failure, give every
rejection a stable machine-matchable diagnostic, expose the import knobs users
actually need, and back the performance and compatibility claims with recorded
measurements instead of expectations.

## What shipped

- **Large-asset fix:** the v0.1.0 read path serialized the generated stage to
  USDA text and reparsed it, which failed at multi-million-Gaussian sizes. The
  layer is now authored directly and transferred without a text round-trip;
  the 5.83M-splat Mip-NeRF 360 `garden` reference opens in 16.4 s with a
  2.39 GiB peak on the baseline machine.
- **Read-path memory reduction:** properties decode directly to `float` and
  every intermediate representation is released as soon as the next one is
  built. On a 696k-Gaussian SH-degree-3 capture this cut peak commit memory
  from 8.05 GiB to 0.40 GiB and `Usd.Stage.Open` from 15.3 s to 1.9 s.
- **Metadata-only reads:** `Read(metadataOnly=true)` authors the `/Asset`
  scaffold, stage metrics, and header-derived Gaussian count and SH degree
  without decoding vertex data — ~5 ms at any asset size.
- **File-format arguments:** `shDegree` (cap the imported SH degree, 0-3),
  `opacityThreshold` (drop Gaussians below a decoded-opacity threshold, 0-1),
  and `scaleMultiplier` (rescale linear scales, > 0), each validated with
  stable diagnostics.
- **Stable diagnostics:** every error and warning starts with a
  `[GSPLY-****]` code that is never renumbered or reused, with a
  machine-readable catalog shipped in the plugin resources
  (`diagnostics.json`) and cross-checked by tests in both directions.
- **Observed dialect compatibility**
  ([PLY_DIALECTS.md](../reference/PLY_DIALECTS.md)): Graphdeco reference
  (`garden`, 5.83M Gaussians), Brush v0.3.0 (lexicographic `f_rest` order),
  and Postshot all decode identically; property resolution is name-based, so
  declaration order never matters. Unsupported input is enumerated and fails
  with a stable diagnostic.
- **Performance baselines**
  ([PERFORMANCE_BASELINES.md](../reference/PERFORMANCE_BASELINES.md)):
  design-policy §12.1 measurements across six assets from 3 to 5.83M
  Gaussians, taken with `tools/benchmark_import.py`.
- **Fixture and corpus depth:** degree-2/3 SH fixtures with exact coefficient
  assertions, an asymmetric multi-Gaussian fixture, a scrambled
  property-order fixture, malformed cases for every header-layout diagnostic,
  and two CC0-1.0 author-captured 8,192-Gaussian corpus assets with recorded
  provenance.
- **Reproducible Windows packaging:** MSVC compiles, archives, and links with
  `/Brepro`, removing wall-clock timestamps from objects, library members,
  the PE header, and the debug directory (observed across hosted runs — see
  below).
- **Release-version single-sourcing:** `scripts/release.py set-version`
  rewrites every declared version location, and the release guard fails on
  drift.

Granular detail: [CHANGELOG 0.2.0](../../CHANGELOG.md) and the
[delivery history](../reports/delivery-history.md).

## Compatibility

Unchanged from v0.1.0:

- **OpenUSD:** cy2026 / OpenUSD 26.05 (the tolerated range is `>=26.05,<27.0`;
  the `ParticleField3DGaussianSplat` schema requires 26.05)
- **Platforms:** Windows x86_64 (msvc143), macOS arm64 (libcxx),
  Linux x86_64 (libstdcxx)
- **Python ABI:** 3.13
- **Built with:** OST 0.18.0, same pinned runtimes as v0.1.0

A binary target must match the host OpenUSD build, platform/compiler ABI, and
Python ABI.

**Reproducibility, precisely.** The within-run gate passed on all three cells:
each packages the same build twice and fails if the two `archive_digest`
values disagree. Beyond that, this release adds an *across-run* observation:
the pre-tag dry run (run 29684898495) and the tag run (run 29684959129), both
building commit `7323f46` on separate hosted runners, produced **byte-identical
`tar.zst` archives on Windows and Linux**. For Windows this is the first hosted
confirmation of the `/Brepro` fix — the v0.1.0 archives differed across runs.
The macOS archive still differs across runs (suspected Mach-O
`LC_UUID`/timestamp analog; tracked in the roadmap). The manifest sidecars
differ across runs only in build provenance (`git_ref`, trigger event,
creation time), which is by design; the archives are the reproducibility
surface.

## Known limitations at this release

- Read-only import, and a full read still materializes the entire asset;
  there is no streaming. Metadata-only reads cover the inspection case
  without decoding.
- Canonical Graphdeco property names only; no aliases. SuperSplat
  **compressed** PLY, `binary_big_endian`, and non-Gaussian PLY are rejected
  with stable diagnostics
  ([known unsupported cases](../reference/PLY_DIALECTS.md#known-unsupported-cases)).
- No bundled Gaussian renderer — this project authors Gaussian USD data. A
  stage opens even when the active Hydra renderer cannot draw splats.
- Windows verified to OST L4 rather than L5 (see above).
- Package-origin L5 still **skips rather than executes**; the upstream OST
  packaging seam recorded in
  [dogfooding report 01](../reports/ost/01-2026-07-18-v0.18.0-bootstrap.md)
  is unchanged in OST 0.18.0.
- macOS package archives are not yet digest-reproducible across runs.
- SPZ, glTF/GLB Gaussian extensions, and SOG remain out of scope; SPZ is the
  v0.3.0 theme in the [release plan](../roadmap/release-plan.md).
