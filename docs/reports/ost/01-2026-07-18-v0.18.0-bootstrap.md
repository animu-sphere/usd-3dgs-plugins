# OST dogfooding report #1 — empty repository to a packaged Gaussian PLY file-format plugin

- Date: 2026-07-18
- OST: 0.18.0
- Host: Windows x86_64, MSVC 14.34 / toolset 143, CMake 4.4.0-rc2,
  Ninja 1.13.2
- Runtime: `openstrata-cy2026-windows-x86_64-py313-usd`, OpenUSD 26.05,
  CPython 3.13, source-built and validated
- Context: create `usd-3dgs-plugins` from an empty Git repository while using
  `usd-vrm-plugins` as the OpenStrata workspace reference.

This is the first report in this repository's append-only OST dogfooding series.
It covers scaffold, a manifest-declared plain-library edge, real C++ plugin
build, L0-L5 source verification, CI generation, packaging, notices, and
package-origin verification.

## TL;DR

OST 0.18.0 carried the initial project from empty repository to a real packaged
OpenUSD plugin without a repository-owned environment launcher or dependency
bootstrap script. The important workspace path worked:

```text
gaussian-ply -> gaussianCore
```

`ost plugin build` built and installed the plain library before configuring the
consumer; `ost plugin test --workspace` reported and validated that edge; and
`ost plugin package` staged the dependency closure and declared third-party
notice.

The local result is green through source L5. The package-origin result is green
through L4 and reports one L5 skip because the package contains the roundtrip
fixture but not the adjacent golden file. That is the only concrete upstream
product seam found in this first pass.

## 1. Commands exercised

The project was created with:

```console
ost init --name usd-3dgs-plugins --platform cy2026 --template usd-plugin-workspace
ost plugin new usd-fileformat gaussian-ply --extension ply --dir plugins/gaussian-ply
```

The final verification sequence was:

```console
ost plugin build plugins/gaussian-ply --json
ctest --test-dir libs/gaussian-core/build/cy2026-windows-x86_64-py313-usd --output-on-failure

ost plugin test --workspace --up-to 5 --json
ost plugin package plugins/gaussian-ply --json
ost plugin test plugins/gaussian-ply --from-package --up-to 5 --json
ost ci validate --json
ost ci generate github --json

  "C:\Users\snkm\Desktop\testdata_3dgs\3DGS_PLY_sample_data\PLY(postshot)\cactus_splat3_30kSteps_142k_splats.ply"
```

## 2. What worked well

### 2.1 Scaffold to real plugin

The workspace and `usd-fileformat` templates provided the correct contract
surfaces immediately: `openstrata.toml`, bundle manifest, generated
`plugInfo.json.in`, CMake helper, fixtures, and standard output locations. The
scaffold was designed to be replaced by a real parser rather than forcing a
second repository layout.

### 2.2 Plain-library closure

`gaussianCore` is described by `openstrata.library.yaml` and consumed as:

```yaml
requires:
  libraries:
    - id: gaussianCore
      version: ">=0.1,<0.2"
```

OST built the library, installed its headers/static archive/CMake package into
the workspace prefix, then configured and linked `gaussian-ply`. The workspace
test reported the exact source-workspace provenance and CMake target
`openstrata::gaussianCore`. No sibling `add_subdirectory` workaround was needed
inside the bundle.

### 2.3 Real OpenUSD schema path

The runtime contained OpenUSD 26.05's
`UsdVolParticleField3DGaussianSplat`. Discovery, `usdcat`, and
`Usd.Stage.Open()` all exercised the plugin's real `SdfFileFormat` entry point;
the test was not a parser-only substitute.

### 2.4 Package notices

Adding a bundle-relative `notices` entry copied `THIRD_PARTY_NOTICES.md` into the
canonical package root. The resulting archive contained the notice, manifest,
plugin DLL, fixtures, and dependency closure without a hand-authored package
file list.

### 2.5 CI generation and evidence refresh

The matrix reuses the reference workspace's immutable Windows/macOS/Linux
runtime and OCI pins. The first `ost ci validate` found that the locally cached
Windows artifact predated attached evidence. Pulling the same pinned OCI
manifest with `--expect-artifact` returned the same artifact digest, verified
SBOM and provenance, and refreshed the evidence view; validation then passed
with three cells and no gaps. This was good recovery behavior and preserved the
artifact pin.

### 2.6 Dual-mode root build

The non-OST root path also configured successfully against the same OpenUSD
runtime with Visual Studio 2022. Dynamic bundle discovery found
`plugins/gaussian-ply`; the build composed `gaussianCore` first and all three
root CTest entries passed. This verifies that the root is a real fallback build,
not documentation-only scaffolding.

## 4. Upstream asks and observations

### P1 — make the source golden part of package-origin L5

The source bundle has:

```text
tests/fixtures/one-gaussian-ascii.ply
tests/fixtures/one-gaussian-ascii.ply.golden.usda
```

The manifest declares the PLY as a roundtrip fixture. `ost plugin package`
copies the declared fixture but not the adjacent golden, and the manifest has no
field for a golden path. Consequently the command explicitly requested with
`--up-to 5` can never execute L5 from this package even though source L5 passes.

Any of these would close the seam:

1. automatically package `<roundtrip-fixture>.golden.usda` when it exists;
2. add an explicit roundtrip fixture/golden pair to the manifest;
3. make package-origin L5 generate and compare against a packaged canonical
   representation recorded in the artifact manifest.

Acceptance test:

```text
ost plugin test <bundle> --from-package --up-to 5
-> golden.roundtrip PASS, not SKIP
```

### P2 — explain the evidence-gap recovery command

`ost ci validate --json` accurately reported that the locally present Windows
runtime artifact lacked SBOM/provenance evidence. Because the matrix already
contained an immutable `runtime_remote` and expected OCI digest, a suggested
action containing the exact safe `ost artifact pull ... --expect-artifact ...`
command would make the successful recovery path discoverable.

### Observation — localized MSVC include output is very noisy

On Japanese Windows, incremental compilation prints a large number of localized
`/showIncludes` lines during `ost plugin build --json`. The build succeeds and
the final JSON is correct, so this is not a blocker. It makes CI logs and
machine-output capture substantially noisier and is worth checking on hosted
Windows before filing a stronger issue.
